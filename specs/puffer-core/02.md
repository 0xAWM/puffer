# puffer-core update 02

## Purpose

Permission handling in `puffer-core` now uses typed internal permission models
layered over legacy config and session inputs for session grants, Browser
approval scope, request-scoped tool filters, and runner-facing filesystem
execution policy.

The goal is to preserve the existing external permission UX and config surface
while making the internal runtime model precise enough to support:

- whole-tool session approvals
- Browser action-scoped and cross-session approvals
- request-scoped filesystem and process tool selectors
- typed filesystem policy at the runner boundary

## Design

`AppState` keeps a typed `SessionPermissionState` alongside legacy session
permission fields. `AllowSession` writes into typed grants first, then projects
the result back into `session_tool_permissions` through a compatibility
projection used by current UI and worker synchronization paths.

`permissions::profile` builds `EffectivePermissionProfile` from legacy
workspace permission settings plus typed session grants. The profile includes:

- tool-level session overrides
- surface-level session grants
- category-level session grants for Browser and cross-session workflow access
- request-scoped tool filters with typed constraints
- future-ready path-prefix grant storage

Browser permission handling is modeled explicitly:

- action categories: `inspect`, `navigate`, `interact`, `evaluate`
- root Browser session resolution from `sessionId`
- a separate cross-session Browser approval category

`permissions::execution` derives typed executor-facing policy from the
effective profile instead of forcing lower layers to infer behavior from legacy
config strings. The derived layer currently exposes:

- filesystem policy
- process policy
- network policy
- a compatibility bridge for older in-process call sites

Request-scoped tool filters keep a typed distinction between:

- filesystem path-pattern constraints
- process command-prefix constraints

Runner-facing filesystem execution uses a typed transport shape:

- `puffer-runner-api::FilesystemExecutionPolicy`
- `puffer-runner-api::FilesystemSandboxMode`

This keeps the runner boundary small while preserving the same sandbox semantics
used by in-process filesystem tools.

## Logic

`AllowAllSession` behavior is unchanged and still bypasses tool-level deny rules
for the active session.

`AllowSession` remains behaviorally compatible for most tools because each
grant still records a legacy whole-tool allow projection, but it can also
record richer typed grants:

- Browser approvals record action categories and cross-session access
- `SendMessage` bridge targets record cross-session workflow access

Browser session approval is intentionally narrower than a whole-tool allow:

- `inspect` is allowed by default for the current Browser session
- `navigate` and `interact` require approval
- `evaluate` requires explicit approval because it executes page JavaScript
- cross-session Browser access requires explicit approval

`AllowSession` for Browser records category grants instead of projecting a
synthetic whole-tool `browser = allow` entry into legacy session permissions.
This preserves Browser approval scope while keeping compatibility projections
for other tools.

Surface grants are retained as part of the typed session-grant model so the
effective profile can summarize session-granted permission surfaces and derive
surface-level runtime state. They are not the approval-bearing source for
Browser allow decisions; Browser allow checks are still based on category grants
or explicit whole-tool overrides.

Request-scoped tool filter behavior is compatibility-focused at the external
selector syntax boundary:

- selectors without parentheses allow the whole tool
- filesystem selectors preserve raw pattern strength and evaluate with
  `glob::Pattern`
- `~` and `~/...` expand against `$HOME`
- concrete file inputs are normalized against the current working directory
- `Bash(...:*)` and `PowerShell(...:*)` match whitespace-token command
  prefixes against the `command` field rather than raw string prefixes

Permission evaluation is split cleanly:

- evaluator logic decides tool visibility plus allow/ask/deny enforcement
- Browser decisions use the same core profile for model-visible filtering and
  call-time approval
- execution derivation maps the effective profile into executor-facing policies
- runner dispatch consumes typed filesystem policy directly
- older in-process surfaces may still consume a compatibility bridge

After a session approval mutation, runtime permission pre-resolution reloads the
effective context through a shared helper so updated derived policy takes effect
immediately on paths that call `resolve_tool_permission()`. The direct
`execute_tool_call()` path uses the permission context loaded before prompting,
so its current-turn execution continues with that pre-prompt snapshot and only
later turns observe the refreshed context.

## Compatibility

External workspace config and permission UX are unchanged:

- `permissions.toml` format stays the same
- legacy tool keys remain accepted as compatibility inputs
- `AllowSession` and `AllowAllSession` keep existing user-visible behavior

Explicit workspace Browser overrides such as `browser = allow`, `ask`, or
`deny` still apply globally. `auto` and `on-request` defer to the typed Browser
taxonomy instead of flattening all Browser calls into one behavior.

Request-scoped selector syntax is unchanged. Parenthesized selectors keep their
Claude-compatible external meaning while now being evaluated through typed
internal constraints instead of lossy prefix-only matching.

Runner dispatch no longer reconstructs filesystem access from
`allow_all_paths`. Typed filesystem policy must be present on the runner
request.

No daemon Browser execution entrypoints change in this update.

## Contracts

`SessionPermissionState` is the source of truth for session grants. Legacy
`session_tool_permissions` remains a compatibility projection derived from
typed grants.

`EffectivePermissionProfile` is a typed internal model over legacy workspace
config, legacy sandbox settings, and compatibility session fields. This update
does not yet make the permission stack end-to-end typed.

`AppState::sync_session_permission_state()` is the inverse compatibility path:
it rebuilds typed session state from the legacy `session_allow_all` flag plus
`session_tool_permissions`. Callers must update `session_allow_all` before
invoking that rebuild helper because the typed `allow_all_tools` bit is
sourced from the legacy flag rather than inferred from tool-level grants.

Browser category grants must not reintroduce a synthetic `"browser" -> "allow"`
legacy projection during session-state round trips.

Browser evaluate approvals granted for the session must survive worker-to-TUI
synchronization and later turns without re-prompting.

Model-visible Browser filtering and call-time Browser approval must consume the
same effective profile decision so visibility and execution stay aligned.

Surface-level session-granted state is descriptive runtime metadata. New allow
decisions must not rely only on `surface_grants` or derived `session_granted`
flags without an explicit permission-model change and matching regression
coverage.

## Read-Path Audit

Current `surface_grants` and derived `session_granted` consumers fall into
three buckets:

- approval-bearing consumers
- descriptive runtime-state consumers
- UI/session-state consumers

Approval-bearing consumers are the only paths that may directly determine
allow-versus-ask behavior for Browser calls. At present, Browser approval is
derived from:

- Browser category grants
- explicit whole-tool Browser overrides

Descriptive runtime-state consumers summarize typed session permission state
without being approval authority. These currently include:

- `SessionGrantProfile.surface_grants`
- `EffectiveSurfaceProfile.session_granted`
- derived executor-facing `session_granted` fields

UI/session-state consumers expose whether the session has accumulated a surface
grant without changing permission outcomes. These currently include:

- `SessionPermissionState::has_browser_grant()`

This distinction is intentional and must remain explicit. `surface_grants` and
derived `session_granted` flags may be used for profile summaries, executor
metadata, or UI state, but they must not become approval-bearing implicitly.
Any future change that wants to treat them as allow authority must update the
permission model contract and add targeted regression coverage showing that the
broadened semantics are intentional.

Worker-to-UI synchronization should prefer replacing the full typed session
state when it is available. That preserves category-level grants across the
round trip and avoids depending on the lossy inverse compatibility path or its
`session_allow_all` ordering requirement for correctness.

## Tests

Permission and runner regression coverage now asserts that:

- session grants keep the legacy whole-tool allow projection for non-Browser
  tools
- Browser session grants are scoped to action category and cross-session access
- Browser category grants do not reintroduce legacy `"browser" -> "allow"`
  projection
- a Browser surface-only session grant does not allow `Browser evaluate`
- Browser evaluate session approval survives session-state round trips
- request-scoped `Bash(prefix:*)` selectors allow matching commands and deny
  non-matching commands, including token-boundary cases such as rejecting
  `Bash(npm:*)` for `npmx install`
- request-scoped filesystem selectors expand `$HOME`
- request-scoped filesystem selectors preserve non-prefix glob strength such as
  `*.rs` and `**`
- workspace-write runner requests reject out-of-workspace filesystem reads
- danger-full-access runner requests allow them
